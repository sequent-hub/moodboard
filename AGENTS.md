# Правила Для Агентов

## Стек

- Пакет: `@sequent-org/moodboard` — интерактивный редактор мудборда, поставляется как ESM-библиотека (`type: module`, точка входа `src/index.js`, публичный класс `MoodBoard`).
- Рендер: `pixi.js` 7 (canvas/WebGL) + HTML-оверлеи поверх canvas (текст, ручки).
- Сборка/dev: `vite` 5 (без бандла либы — публикуется исходник). Dev-прокси `/api` -> `localhost:3001`.
- Сеть: `axios`. Иконки: `lucide-static`.
- Тесты: `vitest` + `jsdom` + `@testing-library/*` (юнит/контракт), `@playwright/test` (e2e).
- Runtime: браузер; Node только для скриптов/деплоя. Бэкенд — Laravel (см. `LARAVEL_USAGE.md`).

## Структура

- `src/index.js` — экспорт пакета.
- `src/moodboard/` — Facade и оркестрация: `MoodBoard`, `WorkspaceManager`, `DataManager`, `ActionHandler`; `bootstrap/`, `integration/`, `lifecycle/`.
- `src/core/` — ядро: `index.js` (`CoreMoodBoard`), `EventBus`, `StateManager`, `HistoryManager` (Command/Undo-Redo), `SaveManager`, `ApiClient`, `PixiEngine`; `events/Events.js` (реестр событий), `commands/`, `flows/`, `rendering/`, `keyboard/`, `bootstrap/`.
- `src/objects/` — типы объектов + `ObjectFactory` (Frame, Shape, Drawing, Text, Emoji, Image, Comment, Note, File).
- `src/tools/` — инструменты ввода: `ToolManager`, `BaseTool`, `board-tools/` (pan/zoom), `object-tools/` (select/placement/text/drawing), `object-tools/selection/` (resize/rotate/group-контроллеры).
- `src/ui/` — UI: `Toolbar`, `Topbar`, `ZoomPanel`, `MapPanel`, `ContextMenu`, `*PropertiesPanel`, `HtmlTextLayer`, `HtmlHandlesLayer` (активная система ручек), `handles/`, `chat/` (AI-чат), `text-properties/`, `styles/`.
- `src/services/` — бизнес-логика: `BoardService`, `ZoomPanController`, `ZOrderManager`, `FrameService`, `Image/FileUploadService`, `SettingsApplier`, `GridSnapResolver`, `ai/`.
- `src/grid/` — сетка: `GridFactory`, `Base/Dot/Line/CrossGrid`, `ScreenGridPhaseMachine` (screen-space рендер).
- `src/mindmap/`, `src/utils/`, `src/assets/`, `public/` — mind-map контракт, утилиты, шрифты/иконки/emoji, статика.
- `tests/` (core/tools/ui/objects/services/integration), `scripts/` (диагностика grid, screen-integer контракт).

## Пин-каноны

- Архитектура event-driven: подсистемы общаются через `EventBus`; имена событий — только из `src/core/events/Events.js`, payload-контракты не менять без явной задачи.
- Паттерны: Facade (`MoodBoard`), Factory (`ObjectFactory`), Command (`core/commands/*` + `HistoryManager`).
- Координаты: `state.position` хранится как top-left, PIXI работает от center. Конвертация: `x = left + w/2`, `y = top + h/2`. Детали и риски — `COORDINATE_SYSTEM.md`.
- PIXI — версия `pixi.js` 7; не дрейфуй в API v8 (`new Application({...})`/`app.view`/`beginFill…endFill`/`eventMode`), снимай listeners и текстуры перед `destroy`. Канон — `.cursor/rules/pixi-canon.mdc`.
- Pixel-perfect integer contract: в screen-space запрещены дробные px; `x/y/width/height/step/radius/line positions` — integer; округление в точке формирования screen-space значений. Проверка: `npm run test:screen:integer`.
- Перед/после рефактора больших файлов — baseline-тесты поведения (см. скрипты `test:*:baseline`).
- Lifecycle: сохранять корректные `attach/detach/activate/deactivate/destroy`, не плодить listeners.
- Подробнее: `ARCHITECTURE.md`.

## Назначение

Эти правила защищают проект от раздувания файлов, смешения ответственностей и рискованных рефакторов.

## Правила По Размеру Файлов

- До `400` строк: допустимо, если у файла одна понятная ответственность.
- `400-600` строк: перед добавлением нового кода нужно проверить, не смешаны ли в файле разные ответственности.
- `600+` строк: нельзя дальше расширять файл без объяснения, почему новый файл хуже.
- `800+` строк: обязательный кандидат на рефактор.
- `1000+` строк: приоритетный кандидат на рефактор.

Количество строк является только эвристикой. Границы ответственности важнее, чем сам размер файла.

## Правила По Ответственности

- Один файл должен иметь одну основную ответственность.
- Новый функционал по умолчанию нужно выносить в новый файл, а не дописывать в уже большой файл.
- При разделении предпочитать сфокусированные модули: `Controller`, `Service`, `Renderer`, `Factory`, `Flow`, `Bridge`, `Store`.
- Не смешивать в одном файле UI-рендеринг, wiring событий, DOM-логику, управление состоянием и бизнес-правила.

## Правила Рефактора

- Сначала выполнять механическое разделение больших файлов без изменения поведения.
- Нельзя совмещать архитектурный рефактор и изменение функционального поведения в одной задаче.
- Во время стабилизационного рефактора сохранять имена событий, контракты payload, публичные API и критичный порядок вызовов.
- После стабилизации допускаются последующие рефакторы для улучшения границ модулей.

## Правила Тестирования

- Перед рефактором большого файла нужно создать baseline-тесты текущего поведения.
- После рефактора нужно прогнать те же тесты для подтверждения эквивалентности поведения.
- Предпочитать тесты поведения и контрактов, а не тесты приватной реализации.
- Для screen-space изменений обязательно прогонять `npm run test:screen:integer`.

## Pixel-Perfect Integer Contract

- В screen-space запрещены дробные пиксели (`.5`, `13.2`, любые non-integer значения).
- `x/y/width/height/step/radius/line positions` в экранных координатах должны быть integer.
- Округление выполнять в точках формирования screen-space значений.

## Безопасность Событий И Listener-ов

- После рефакторов нужно проверять, что listeners не дублируются.
- Нужно сохранять корректный lifecycle cleanup для `attach`, `detach`, `activate`, `deactivate`, `destroy`.
- Контракты EventBus должны оставаться стабильными, если задача явно не требует их изменить.

## Правила Рабочего Процесса

- Рискованные рефакторы выполнять небольшими этапами.
- Если пользователь просит, работать по правилу `один шаг = один файл`.
- Если файл уже большой, нужно объяснить, почему код добавляется в него, а не выносится в новый модуль.
- Если есть сомнение, выбирать вариант, который уменьшает связность и локализует будущие изменения.
- Команда `gp`: если пользователь пишет в чате `gp`, нужно безопасно синхронизировать локальную и удаленную ветки.
- Перед синхронизацией проверить текущую ветку, `git status` и наличие локальных незакоммиченных изменений.
- Если есть локальные незакоммиченные изменения, сохранить их безопасно (например, через `git stash push -u` или эквивалентный безопасный шаг), чтобы ничего не потерялось.
- Проверить удаленную ветку (`git fetch`) и, если там есть новые коммиты, корректно объединить их с текущей локальной историей (через `rebase` или `merge` по ситуации) с разрешением конфликтов без потери данных.
- После объединения гарантированно убедиться, что и локальные, и удаленные изменения сохранены в локальной ветке; затем восстановить локальные сохраненные изменения, если они были отложены.
- Выполнить финальную проверку `git status`, убедиться в целостности результата и только после этого сделать `git push` в удаленный репозиторий.
- Целевое состояние после `gp`: локальная ветка наиболее актуальна относительно удаленной, и итоговые изменения отправлены в remote без потери локальных правок.
- Команда `gc`: если пользователь пишет в чате `gc`, нужно создать новый commit текущих изменений.
- Перед commit обязательно повысить patch-версию пакета в `package.json` (пример: `1.4.84` -> `1.4.85`), и это изменение версии должно входить в тот же commit.
- При каждом commit через `gc` всегда включать изменение версии пакета вместе с остальными актуальными изменениями.
- Формат commit message для `gc`: начинать с типа в виде `feat(note):`, далее писать текст на русском языке.
