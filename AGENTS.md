# Правила Для Агентов

## Назначение

Эти правила защищают проект от раздувания файлов, смешения ответственностей и рискованных рефакторов.

## Правила По Размеру Файлов

- До `400` строк: допустимо, если у файла одна понятная ответственность.
- `400-600` строк: перед добавлением нового кода нужно проверить, не смешаны ли в файле разные ответственности.
- `600+` строк: нельзя дальше расширять файл без объяснения, почему новый файл хуже.
- `800+` строк: обязательный кандидат на рефактор.
- `1000+` строк: приоритетный кандидат на рефактор.

Количество строк является только эвристикой. Границы ответственности важнее, чем сам размер файла.

## Правила По Ответственности

- Один файл должен иметь одну основную ответственность.
- Новый функционал по умолчанию нужно выносить в новый файл, а не дописывать в уже большой файл.
- При разделении предпочитать сфокусированные модули: `Controller`, `Service`, `Renderer`, `Factory`, `Flow`, `Bridge`, `Store`.
- Не смешивать в одном файле UI-рендеринг, wiring событий, DOM-логику, управление состоянием и бизнес-правила.

## Правила Рефактора

- Сначала выполнять механическое разделение больших файлов без изменения поведения.
- Нельзя совмещать архитектурный рефактор и изменение функционального поведения в одной задаче.
- Во время стабилизационного рефактора сохранять имена событий, контракты payload, публичные API и критичный порядок вызовов.
- После стабилизации допускаются последующие рефакторы для улучшения границ модулей.

## Правила Тестирования

- Перед рефактором большого файла нужно создать baseline-тесты текущего поведения.
- После рефактора нужно прогнать те же тесты для подтверждения эквивалентности поведения.
- Предпочитать тесты поведения и контрактов, а не тесты приватной реализации.
- Для screen-space изменений обязательно прогонять `npm run test:screen:integer`.

## Pixel-Perfect Integer Contract

- В screen-space запрещены дробные пиксели (`.5`, `13.2`, любые non-integer значения).
- `x/y/width/height/step/radius/line positions` в экранных координатах должны быть integer.
- Округление выполнять в точках формирования screen-space значений.

## Безопасность Событий И Listener-ов

- После рефакторов нужно проверять, что listeners не дублируются.
- Нужно сохранять корректный lifecycle cleanup для `attach`, `detach`, `activate`, `deactivate`, `destroy`.
- Контракты EventBus должны оставаться стабильными, если задача явно не требует их изменить.

## Правила Рабочего Процесса

- Рискованные рефакторы выполнять небольшими этапами.
- Если пользователь просит, работать по правилу `один шаг = один файл`.
- Если файл уже большой, нужно объяснить, почему код добавляется в него, а не выносится в новый модуль.
- Если есть сомнение, выбирать вариант, который уменьшает связность и локализует будущие изменения.
