# Как определить, есть ли метаданные в скриншоте

Ниже практический способ проверки для кейса, когда данные встраиваются в пиксели изображения (LSB-стеганография) и оборачиваются маркерами:

- начало: `START_TEXT_`
- конец: `_END_TEXT`

## 1) Что считать признаком метаданных

Для такого формата метаданные считаются найденными, если одновременно выполняются условия:

1. В извлеченном потоке символов есть `START_TEXT_`.
2. В потоке есть `_END_TEXT`.
3. `_END_TEXT` идет после `START_TEXT_`.
4. Между маркерами есть непустая строка.

Если хотя бы одно условие не выполняется — считаем, что метаданные не обнаружены.

## 2) Короткий алгоритм проверки

1. Открыть изображение и прочитать RGBA-пиксели.
2. Для каждого пикселя получить байт из младших битов RGB:
   - `byte = (r & 0x07) | ((g & 0x07) << 3) | ((b & 0x03) << 6)`
3. Преобразовать байты в символы и собрать строку.
4. Найти в строке маркеры `START_TEXT_` и `_END_TEXT`.
5. Вырезать участок между маркерами.
6. Декодировать участок как UTF-8 (если требуется).
7. Вернуть результат проверки: найдено/не найдено + полезная строка.

## 3) Минимальный пример (JavaScript, browser)

```js
async function extractHiddenTextFromImage(fileOrBlob) {
  const bitmap = await createImageBitmap(fileOrBlob);
  const canvas = document.createElement('canvas');
  canvas.width = bitmap.width;
  canvas.height = bitmap.height;

  const ctx = canvas.getContext('2d', { willReadFrequently: true });
  ctx.drawImage(bitmap, 0, 0);

  const { data } = ctx.getImageData(0, 0, canvas.width, canvas.height); // RGBA
  let stream = '';

  for (let i = 0; i < data.length; i += 4) {
    const r = data[i];
    const g = data[i + 1];
    const b = data[i + 2];

    const byte = (r & 0x07) | ((g & 0x07) << 3) | ((b & 0x03) << 6);
    if (byte === 0) continue;
    stream += String.fromCharCode(byte);
  }

  const startToken = 'START_TEXT_';
  const endToken = '_END_TEXT';
  const start = stream.indexOf(startToken);
  const end = stream.indexOf(endToken, start + startToken.length);

  if (start === -1 || end === -1 || end <= start) {
    return { hasMetadata: false, payload: null };
  }

  const raw = stream.slice(start + startToken.length, end);
  if (!raw.length) {
    return { hasMetadata: false, payload: null };
  }

  // Если нужно именно UTF-8 из байтов:
  const bytes = Uint8Array.from([...raw].map(ch => ch.charCodeAt(0)));
  const payload = new TextDecoder('utf-8', { fatal: false }).decode(bytes);

  return { hasMetadata: true, payload };
}
```

## 4) Диагностика (рекомендуется)

Чтобы надежно понять, есть ли метаданные в конкретном файле, добавьте логирование:

- размер изображения (`width x height`);
- число обработанных пикселей;
- позиция `start` и `end` маркеров;
- длина полезной строки между маркерами;
- первые 80 символов извлеченного payload (без чувствительных данных).

Пример диагностического вывода:

```text
[META-CHECK] image=1920x1080 pixels=2073600
[META-CHECK] start=1542 end=1890 found=true payload_len=339
[META-CHECK] payload_preview={"camera":{"x":...},"guid":"..."}
```

## 5) Важные ограничения

- JPEG может менять пиксели при пересохранении, из-за чего скрытые данные могут повреждаться.
- После компрессии/ресайза/скриншота "скриншота" маркеры могут не находиться.
- Для воспроизводимой проверки используйте оригинальный файл без промежуточной обработки.

## 6) Критерий "годен для интеграции"

Скриншот считаем пригодным, если:

- `hasMetadata = true`;
- payload успешно декодируется;
- payload проходит вашу доменную валидацию (например, JSON.parse + проверка обязательных полей).

## 7) Как работает переход по клику "Показать в модели" (карточка -> Revit)

Ниже описана фактическая схема взаимодействия, которую можно повторить в другом проекте.

### 7.1. Точка входа в UI

1. В карточке показывается ссылка `Показать в модели` только если у вложения заполнено поле `view`.
2. Поле `view` записывается при загрузке вложения и хранит payload (метаданные вида/позиции/контекста).
3. На ссылку навешивается обработчик клика (`openViewButton`).

### 7.2. Что делает клик

По клику фронтенд:

1. Берет строку payload из `data-view`.
2. Перебирает локальные порты `11210..11220` на `127.0.0.1`.
3. На каждый порт отправляет `POST`-запрос, где `body = payload`.

Важно: в этой схеме веб-приложение не запускает Revit напрямую. Оно только отправляет команду в локальный десктопный сервис на localhost.

### 7.3. Где открывается Revit

Открытие Revit происходит на стороне локального приложения (desktop companion), которое:

1. слушает один из портов `11210..11220`;
2. принимает payload из POST-запроса;
3. интерпретирует payload;
4. вызывает действия в Revit (например, открыть модель/перейти к виду/элементу).

### 7.4. Минимальная последовательность (sequence)

```text
Пользователь
  -> Карточка (клик "Показать в модели")
  -> Frontend JS (берет data-view)
  -> POST http://127.0.0.1:11210..11220 (body=payload)
  -> Локальный desktop-сервис (находит свой порт)
  -> Обработка payload
  -> Команда в Revit
  -> Открытие модели/вида в Revit
```

### 7.5. Что логировать для диагностики в другом проекте

Чтобы понять, где цепочка обрывается, логируйте по этапам:

- **Frontend**
  - факт клика;
  - длину payload и первые 80 символов;
  - список портов, куда ушла отправка;
  - HTTP-статус/ошибку для каждого порта.
- **Desktop-сервис**
  - порт, на котором принят запрос;
  - размер body и успешность парсинга payload;
  - результат вызова API/SDK Revit;
  - коды ошибок при неуспехе.
- **Revit-плагин/интеграция**
  - принята ли команда;
  - какой документ/вид открыт;
  - исключения при открытии или навигации.

### 7.6. Критерий успешного перехода

Переход считается успешным, если одновременно есть:

1. клик в UI;
2. успешная доставка payload на один из localhost-портов;
3. подтверждение обработки payload desktop-сервисом;
4. подтверждение выполнения команды в Revit.
