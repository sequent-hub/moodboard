# Универсальные коннекторы MoodBoard (дизайн-спецификация)

> **Статус: ДИЗАЙН, не реализовано.** Документ фиксирует целевую архитектуру коннекторов
> (стрелок/линий с привязкой к объектам, как в Miro/tldraw) до начала кодирования.
> Файлы из раздела 5 пока **не существуют** — это план. Источник механики: открытый
> infinite-canvas SDK `tldraw` (context7 `/tldraw/tldraw`), адаптированный под PIXI-архитектуру
> и инвариант координат проекта (см. `COORDINATE_SYSTEM.md`).

## 1. Базовая идея (из tldraw)

Главный принцип: **привязка (binding) — это данные терминала линии, а не геометрия линии.**
Коннектор хранит два терминала (`start`, `end`); каждый терминал — либо привязка к объекту,
либо свободная точка. Привязка описывается через:

- `boundId` — id целевого объекта;
- `anchor` (= tldraw `normalizedAnchor`) — точка внутри объекта в **долях габаритов** `0..1`
  (`{x:0.5, y:0.5}` — центр, `{x:0, y:0}` — левый верх, `{x:1, y:1}` — правый низ);
- `isPrecise` — `false`: целимся в центр (стабильно при ресайзе); `true`: уважаем `anchor`;
- `isExact` — `false`: конец останавливается на **кромке** объекта; `true`: идёт в точную точку внутрь.

Следствие: при перемещении/ресайзе цели **абсолютные координаты якоря пересчитываются сами**:
`worldAnchor = topLeft + { anchor.x · width, anchor.y · height }`. Это убирает необходимость
хранить и синхронизировать абсолютные координаты концов.

## 2. Модель данных

Коннектор — **полноценный тип объекта** в `state.objects[]` (а не отдельный binding-store).
Причина: так он ложится на существующие `ObjectFactory`, `HistoryManager`, save/load без новых
подсистем хранения.

```javascript
{
  type: 'connector',
  // position/size номинальны (bounding box линии), реальная геометрия — из терминалов
  properties: {
    start: { boundId: 'obj_123', anchor: { x: 0.5, y: 0.5 }, isPrecise: false, isExact: false },
    //   ИЛИ свободный конец:
    // start: { point: { x: 100, y: 200 } },
    end:   { point: { x: 400, y: 250 } },
    style: {
      stroke: 0x2563EB,
      width: 2,
      dash: false,
      head: { start: false, end: true },   // стрелки на концах
      route: 'straight' // 'straight' | 'bezier' | 'elbow'
    }
  }
}
```

Терминал имеет ровно одну из форм: `{ boundId, anchor, isPrecise, isExact }` (привязан)
или `{ point }` (свободный).

## 3. Координаты и инвариант проекта

- Якоря и концы считаются в **world** и рисуются в `worldLayer` → панорама/зум работают
  автоматически, пересчёт в screen не нужен.
- Якорь считается от **top-left** объекта (`state.position` хранится как top-left, см.
  `COORDINATE_SYSTEM.md`): `worldAnchor.x = pos.x + anchor.x · width`.
- Для повёрнутых целей: `worldAnchor` поворачивается вокруг центра объекта на его угол.
- Screen-space integer contract напрямую **не** применяется (рисуем в world), но при отрисовке
  допустимо округление концов в world-координатах, как уже сделано в `MindmapConnectionLayer`.

## 4. Резолвер биндинга (`terminal → world point`)

Ответственность `ConnectorBindingResolver`:

1. **Привязанный терминал, `isPrecise=false`** → целевая точка = центр объекта.
2. **Привязанный терминал, `isPrecise=true`** → целевая точка = `worldAnchor` (раздел 3).
3. **Проекция на кромку (`isExact=false`)**: построить луч от противоположного терминала к целевой
   точке, отсечь по габаритам цели (Liang–Barsky для AABB). Для повёрнутого объекта — перевести
   луч в локальные координаты (минус угол вокруг центра), отсечь по AABB, вернуть обратно.
4. **`isExact=true`** → вернуть целевую точку без отсечения.
5. **Свободный терминал** → вернуть `point` как есть.

## 5. Файлы (план, по правилу «один файл — одна ответственность»)

| Файл | Ответственность |
|---|---|
| `src/objects/ConnectorObject.js` | тип объекта, дефолты, сериализация |
| `src/ui/connectors/ConnectorLayer.js` | рендер всех коннекторов; обобщение `MindmapConnectionLayer`, та же подписка на события перерисовки |
| `src/services/ConnectorBindingResolver.js` | `terminal → world point`, проекция на кромку, поворот цели (раздел 4) |
| `src/tools/object-tools/ConnectorTool.js` | UX: hover-якоря на объектах, drag от объекта к объекту, drop → бинд/свободный конец |

Реестр событий — только из `src/core/events/Events.js` (правило `events-and-lifecycle`). Возможно
понадобится добавить события создания/обновления коннектора через skill `add-event`.

## 6. Целостность связей (требует решения до кода)

При `Events.Object.Deleted` цели нужно поведение для терминалов, ссылающихся на удалённый
`boundId`. Варианты (выбрать один как дефолт):

- **(A) Отвязать**: терминал превращается в свободный `point` в последней world-позиции якоря.
- **(B) Удалить коннектор**: если хотя бы один конец «повис».
- **(C) Каскад tldraw-стиль**: удалить только если оба конца повисли, иначе отвязать.

Открытые вопросы:
- Маршрутизация по умолчанию: `straight` / `bezier` / `elbow`?
- Кластеризация (в tldraw связанные стрелкой объекты двигаются вместе) — нужна или нет?
- Можно ли тянуть коннектор «в пустоту» (свободный конец) или только между объектами?

## 7. События перерисовки

Переиспользовать набор подписок из `MindmapConnectionLayer._attachEvents`
(`Object.Created/Deleted/Updated/StateChanged`, `Tool.Drag*/Resize*/Rotate*/Group*`,
`Tool.PanUpdate`, `UI.ZoomPercent`, `History.Changed`, `Board.Loaded`). Полная перерисовка слоя
приемлема для умеренного числа коннекторов; при росте — точечное обновление + пространственный
индекс (отдельная задача).

## 8. Связь с `MindmapConnectionLayer`

`MindmapConnectionLayer` — частный случай коннекторов: связи **выводятся** из mind-map-меты
(`parentId`/`side`), якоря жёстко на 3 стороны, нет хранимой записи и проекции на кромку. После
ввода универсальных коннекторов mind-map можно (опционально, отдельной задачей) перевести на общий
`ConnectorLayer`, сохранив его частное поведение через стиль/маршрут. До этого — оставить как есть.
